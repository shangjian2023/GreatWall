# BdShield 演示与答辩

本文只维护演示顺序和表达边界。算法、指标和实验数字分别引用 `ARCHITECTURE.md` 与 `EXPERIMENTS.md`，不在这里复制维护。

> 当前竞赛算法实现位于 `competition_core/`。工作台的 `competition_sequence_probe` 只负责调用并展示该目录产生的 sequence mining 与 latent probe 证据，不把算法复制回旧 `src/`。原始归档报告继续固定保存 `INCONCLUSIVE`；竞赛前端依据冻结的概率差 + 候选族支持双条件校准给出直接展示结论。

## 一句话定位

BdShield 是开放权重模型上线前的后门准入审查：在单待审模型上从异常输出构造软触发探测证据，并可选用参考模型追加可读的复核证据。

## 六分钟演示

### 0:00-0:50 供应链问题

说明第三方模型权重无法像源代码一样直接审查，普通能力评测也难以发现只在窄输入条件下激活的隐藏行为。

### 0:50-1:20 方法边界

打开“方法边界”：只审查微调写入开放权重的生成式模型后门，不讨论 prompt injection，也不声称检测闭源 API。

### 1:20-3:40 隐式条件后门检测

1. 点击“新建隐式检测”并选择待审模型；选择器只列出 `competition_runs` 中完成训练的最终 Adapter，隐藏 epoch checkpoint、smoke 与旧实验目录。界面固定使用 `competition_sequence_probe`、`coverage_audit`、`general`、完整词表四分片和 `float16`，这些参数以只读配置栏展示，不伪装成可编辑下拉框；旧单 token、参考辅助和 Oracle 路径均不进入竞赛 GUI。
2. 展示完整 tokenizer 词表如何分成四个区间逐片挖掘，以及四片完成后的候选合并结果。
3. 展示合并候选中的 Top-4 概率探测轨迹，包括候选输出、匹配对照、最大概率差和固定概率判据。
4. 展开一个候选的逐 token 交互：每行核对当时的响应前缀输入、下一 token 输出、概率与
   greedy/beam 选择方式；再展开一个潜变量优化步，核对该批真实 holdout 输入和候选/对照
   两次模型概率输出。说明连续潜变量本身不可读，界面不会把它伪造成自然语言提示词。
5. 展开“新输入白盒回放”：对同一条从未参与优化的问题并列展示无软向量输出和加入已保存
   软向量后的 greedy 输出，再展示完整候选前缀复现率。明确这是连续通道复现，不是可读
   触发器恢复。
6. 展示平均 token 对数似然差，说明它强调低概率 token，目前只作辅助诊断，不参与结论。
7. 展示共享长后缀的候选族支持度与观察门槛，并明确它只是一项开发证据。
8. 最终停在醒目的竞赛检测结论：同一候选的概率差越过 0.25，且候选族支持至少为 5 时显示 `DETECTED · 检测到隐式后门`；没有同时满足时显示 `NOT DETECTED`。原始 JSON 的归档字段保持不变。

当前冻结档案位于 `web/competition-calibration.json`。重新生成时运行
`python -m scripts.build_competition_calibration`，分别通过 `--clean-report` 和
`--backdoor-report` 提供独立报告；阈值只由 clean 队列验收，后门队列只验证冻结规则。

## 常见问题

### 为什么主检测不需要参考模型？

主检测比较的是同一待审模型内，候选输出与匹配良性输出在相同软提示反演预算下的相对吸引性。它仍需要白盒权重、logits 与梯度，并且必须经过独立干净开发模型校准。

### 参考模型还有什么作用？

参考模型用于复核自然语义关联、离散 trigger 和 ASR 分离度。它是增强取证路径，不会被混入无参考 soft-probe 分数。

### 功能性 trigger 不等于训练 trigger，算成功吗？

功能性恢复证明存在可复现的隐藏输入通道；exact match 单独报告，不能作为唯一成功标准。

### 为什么会返回 INCONCLUSIVE？

Stage 1 未召回、Stage 2 未找回或预算不足都无法形成证据闭环。安全场景中必须诚实弃权。

在 `competition_sequence_probe` 中，原始报告的 `INCONCLUSIVE` 是研究归档约束。竞赛 GUI 使用 `gpt2-family-support-dev-v2` 派生 `DETECTED / NOT DETECTED`：概率差只是必要条件，候选族支持负责拦截自然网页记忆造成的概率假阳性。该展示字段不写回原始检测报告。开发集为后门 2/2、clean 0/5，不能冒充冻结后盲测。

## 表达红线

- Oracle、legacy、negative control 和 blind detection 必须明确区分。
- 覆盖审计的 tokenizer 探索仅说明配置覆盖范围，不能讲成任意未知触发器的穷尽检测。
- 不得把概率判据单独讲成 `DETECTED`；只有同候选双条件满足时才能展示校准后的竞赛 `DETECTED`。
- 竞赛模式不接收 clean reference、训练目标、`target_text`、中毒数据或旧 soft-probe calibration；不要把训练侧真值描述成检测输入。
- 历史结果是否使用独立验证问题，以对应 JSON 的 `validation_protocol` 为准。
- 未完成跨架构实测前，不使用“覆盖所有开源模型”“SOTA”或“首创”等表述。
- 不把接口兼容、模型加载成功或单次演示结果外推成泛化能力。

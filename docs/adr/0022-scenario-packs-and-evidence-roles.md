# ADR-0022: 场景包、覆盖审计与 Oracle 取证隔离

- **状态**: Accepted
- **日期**: 2026-07-13
- **相关**: ADR-0016、ADR-0017、ADR-0021

## 背景

竞赛工作台需要让审查者选择通用、代码、金融医疗和中英混排等问题场景，同时不能把某次已知目标输出的复现伪装成盲检，也不能把有限的 tokenizer 探索声称为任意未知 trigger 的穷尽覆盖。

旧平台只有一个扫描入口，报告没有记录问题集、探针策略或结论角色，界面无法可靠地区分这些语义。

## 决策

1. `src/detection/scenarios.py` 定义静态、版本化的场景包。每包有彼此独立的 Stage 1、Stage 2 搜索和留出验证问题集，不含训练 trigger、目标词或攻击配置。
2. 原始报告在 `scan_metadata` 保存 `scan_role`、场景和覆盖凭证，包括问题集数量、输入位置、Stage 1 策略、覆盖焦点与未覆盖范围。
3. 平台提供三类显式角色：
   - `formal_blind`：仅允许已验证的 `general` 场景，沿现役两阶段主路径运行。
   - `coverage_audit`：非通用场景使用 tokenizer 驱动的 `adaptive` Stage 1；属于实验性覆盖审计，报告不得称为穷尽未知 trigger 覆盖。
   - `oracle_diagnostic`：只通过 `/api/oracle-scans` 启动，必须提供 `target_text`，写入 `results/oracle/`，报告固定标为 Oracle，不计入正式盲检成功率。
4. 平台适配器把角色和覆盖凭证作为稳定 schema 的一部分。历史报告缺字段时显示“历史未保存”，不倒推或补造。

## 后果

- 前端可以将正式结论、实验性覆盖和已知目标复现放到同一工作台，但其标签、报告目录和限制说明不可混淆。
- 领域场景目前只改变预注册问题集，不构成跨领域检测有效性的实证声明；正式泛化仍须遵循 `EXPERIMENTS.md` 的训练和验收矩阵。
- Oracle 仍复用 Stage 2 梯度反演和正向验证，以便诊断“Stage 1 未召回”与“Stage 2 失效”，但不改变正式平台扫描入口的盲检约束。

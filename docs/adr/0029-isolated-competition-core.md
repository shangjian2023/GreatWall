# ADR-0029: 隔离式 Competition Core

- **状态**: Accepted
- **日期**: 2026-07-15
- **决策者**: 项目组
- **相关**: ADR-0023、0025、0028

## 背景

旧 `src/` 同时承载历史参考辅助管线、平台适配、实验性候选逻辑与无参考软探测，继续
叠加竞赛实现会扩大耦合和误用训练真值的风险。项目论文实验使用 4×RTX A6000 48 GB，
本地只有单张 RTX 4060 Laptop 8 GB；完整模型矩阵和逐 token 串行扫描无法在竞赛周期内
直接执行。

## 决策

1. 新建 `competition_core/` 作为竞赛算法默认实现，不导入旧 `src/` 或 `scripts/`。
2. 工程术语采用 `sequence_mining`、`latent_probe`、`token_key`、`language_shift`、
   `directive_condition`、`register_condition`，避免代码结构与论文行文一一重复。
3. 训练与检测使用不同 YAML schema。训练目标仅允许出现在训练 YAML 和
   `training_quality_gate`；检测配置与报告不得携带条件、目标文本或中毒数据。
4. 公共数据集严格加载并记录来源指纹、选择索引哈希和内容哈希；加载失败直接终止。
   训练与探测数据按原始 source index 固定分区，fit 与 holdout 不得重叠。
5. 词表首步采用批处理，并支持 token 区间分片与结果合并；长任务可以在单卡上续跑。
6. 连续潜变量探测冻结模型，只优化候选和内部等长对照的软输入。每个更新步骤都检查
   概率差；主阈值为 0.25，首次越过 0.20 的步骤仅作观察。
7. 新报告使用 `criterion_met` 表示单模型方法证据，不借用旧平台的 `DETECTED` 语义。
8. 潜变量探测只接受同方法版本、同检测配置、同待审模型指纹且真值输入全为 false 的
   候选报告；分片合并沿用相同校验。
9. 配置进入潜变量探测的 Top-K 候选必须全部执行，较早候选越过阈值不能截断后续候选；
   报告同时记录执行数、越阈值数和所有候选中的最大概率差。
10. 在不改变论文固定概率判据的前提下，报告候选共享长 token 后缀形成的族支持度。
    该字段用于区分重复强化目标与孤立自然记忆短语，只是开发证据，未经独立 clean
    校准不得提升为正式裁决。
11. 词表挖掘报告使用阶段级配置校验；只修改 probe 预算或开发证据参数时，不要求重新
    执行已经完成且模型指纹、真值隔离标志与 mining 配置均一致的词表扫描。

## 后果

- 竞赛主线可以独立测试、训练和复核，不再受旧平台模块边界影响。
- 单卡吞吐、峰值显存和分片覆盖会进入每份检测报告。
- 旧 `src/` 继续用于历史报告、平台演示与回归，不自动获得新主线的能力声明。
- 在 clean/backdoor 权重和真实检测报告生成前，只能声明工程实现完成，不能声明方法有效。

## 验收

- 检测 YAML 出现训练真值字段时在模型加载前失败。
- 离线测试覆盖严格数据、条件注入比例、分片候选召回与软输入梯度边界。
- `python -m ruff check competition_core` 和 `python -m pytest competition_core/tests -q`
  必须通过。
- 至少一个后门模型和一个匹配 clean 模型使用同一检测配置完成后，结果才可进入竞赛材料。
